6분기 시나리오

🛑 architect 모드1에서 차단 이슈 · spec-writer로 회귀하여 스펙 갱신. developer는 호출되지 않음.
⚠️ developer가 빌드/타입체크 실패 · 원인이 자명하면 직접 수정. 불명이면 debugger 호출. debugger는 분석 → 승인 요청 → 적용.
⚠️ test-writer 실패 · 프로덕션 버그(a) / 테스트 오류(b) / 환경 문제(c)로 분류. (a)면 테스트 수정 금지, developer로 회귀.
🔄 developer 구현 중 구조 결정 막힘 · architect 모드2 호출. 두 안이 비등하면 단순한 쪽을 권고.
📚 N 사이클 누적 · doc-writer가 마스터 플랜·스펙과 코드의 drift를 분류(A/B/C/D)하고, A만 직접 갱신. B/C는 사용자 보고.
🧪 e2e 적용 판정 · 스펙에 E2E 표면이 선언됐고 하니스(e2e:up)가 있으면 실제 엔드포인트 검증. 표면 없으면 N/A. 엔드포인트 흔적은 있는데 미선언이면 e2e-tester가 누락으로 신고(검증 없이 통과되는 갭 방지). 실패는 (a)프로덕션버그/(b)테스트오류/(c)환경문제로 분류 → (a)는 debugger.
🙋 사람 결정 필요 · 보안 수용·신규 의존성·비즈니스·비가역·1모듈 초과 영향은 항상 사람. orchestrator가 결정 레코드 생성 → notifier 발송 → 세션 정지. 사람이 레코드에 답 기록 후 /orchestrate 재실행하면 그 지점에서 재개(resume-on-relaunch).
✅ 사이클 종료 (DoD) · AC 구현 + 테스트 그린 + e2e 통과(또는 N/A) + code-reviewer Critical 0 + security-checker Critical/High 0 + doc-writer 정리 → 작업 브랜치 커밋·PR(기본 브랜치·자동 머지 ❌). 다음 spec-writer로 사이클 반복.

7호출 순서 요약 (체크리스트)

  1. planner → docs/plan/master-plan.md 작성·승인
  2. spec-writer → 사이클 스펙 작성
  3. architect (모드1) → 스펙 리뷰. 차단 이슈 해소까지 ②↔③ 반복
  4. 모드 결정: TDD면 ④ test-writer 선행 / 사후면 ⑤ developer 선행
  5. developer 구현 (필요 시 ③' architect 모드2 / ⑥ debugger 협업)
  6. 사후 모드라면 test-writer 후행, TDD라면 그린 확인
  7. e2e-tester → E2E 표면 판정 + 실제 동작 검증 (DoD)
  8. code-reviewer → 변경사항 리뷰
  9. security-checker → OWASP 심층 검사
  10. N 사이클마다 doc-writer → drift 점검 + 인수인계
  11. DoD 통과 → 커밋·PR, 다음 사이클을 위해 ②로 복귀
자동 구동 · 위 순서는 /orchestrate가 자동으로 dispatch합니다. 사람은 escalation된 결정에만 개입 — 섹션 8.

8⚙️ 오케스트레이션 레이어 (완전 자동화)

위 11개 에이전트를 사람 개입 없이 순차 구동하는 레이어. Claude Code에서 서브에이전트는 다른 서브에이전트를 호출할 수 없으므로, orchestrator는 에이전트가 아니라 메인 세션이 읽는 커맨드 /orchestrate입니다. 프로토콜 단일 진실원천: docs/orchestration.md.

① 결정 라우팅

  1. 에이전트가 못 푸는 결정 → DECISION 레코드 emit
  2. orchestrator가 domain별 판단 에이전트에 라우팅
  3. 불가하면 notifier → 사람
게이트 ▸ 판단주체 ≠ 적용주체

② 완료 게이트(DoD)

  1. AC·테스트·e2e·리뷰·보안·doc 통과
  2. 코드 변경 시 동적 검사 재실행
  3. 통과 후에만 커밋·PR
게이트 ▸ 전부 충족 시 다음 사이클

③ 서킷브레이커·예산

  1. 실패 시그니처 N회 반복 → 트립
  2. 글로벌 예산(dispatch·cycle·시간) 천장
  3. 트립 시 사람에게 escalation
게이트 ▸ .orch-state.json

④ 비동기 사람 응답

  1. notifier 발송 → 결정 영속화
  2. 세션 정지 (대기 ❌)
  3. 답 기록 후 /orchestrate 재실행 → 재개
게이트 ▸ resume-on-relaunch
